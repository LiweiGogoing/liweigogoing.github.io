TCP传输协议理解

发表于 2021-07-11

问题

在学习TCP协议之前，有如下几个疑问：

  1. TCP如何保证传输数据的完整性？
  2. TCP为什么需要3次握手建立连接和4次挥手关闭链接？
  3. TCP的传输过程是怎样的，窗口管理是什么机制？
  4. 当读写主机处理数据速度不一致的时候，数据怎么办，会覆盖数据吗？

概念

TCP是面向链接的，提供点对点全双工的数据传输，是可靠的的字节流传输协议，确保数据传输的完整性。

TCP协议

TCP数据是封装在IP数据报中，通过源IP，目标IP，源端口，目标端口以及协议标识符确定传输通道，IP相关的数据在IP数据的首部，TCP数据分为TCP首部以及TCP数据部分，TCP数据部分是正真传输的数据，都是字节，下面重点分析下TCP首部协议：

字段名 数据大小 释义
源端口 2字节 发送端端口号
目的端口 2字节 接收端端口号
序号 4字节 TCP数据传输过程中的序号
确认号 4字节 对接收到的数据的确认序号
数据偏移 4bit TCP数据部分距离TCP首部的偏移量，就是TCP首部长度
保留 6bit
URG 1bit 紧急标志位置
ACK 1bit 数据确认
PSH 1bit 尽快将数据传到应用层，还不是接收缓冲区
RST 1bit 重新连接
SYN 1bit 请求链接
FIN 1bit 断开链接
窗口 2字节 表示当前主机可以接受的数据大小
校验和 2字节 TCP数据校验值
紧急指针 2字节

一般TCP首部是20字节，可选部分可以填充，下面重点分析其中比较重要的几个字段：

序号：对所有传输的字节流进行编号，确保数据传输连贯性，如果当前的序号为300，携带的数据为100字节，则下一个报文段的序号为400。

确认号：要注意和序号的区分，TCP数据在网络传递过程中可能会出现丢失和重试等，确认号是对收到数据的确认，并且期望下一个传输数据的序号。

数据传输过程中SYN和FIN会消耗一个字节的序号，ACK确认不会消耗序号。后面数据传输过程中会详细介绍序号和确认号工作流程。

窗口：表示报文发送端当前缓冲区剩余大小，单位字节，也就是告诉发送端还能发送多少字节的数据，当窗口为0的时候，发送端就不能发送数据了，需要定时发送探测数据来确认窗口大小。

校验和：网络数据传输过程中可能会出现数据传输出错的情况，通过校验和校验数据，确保数据完整性。

SYN：发起链接，在传输过程中消耗一个字节的序号。

ACK：确认收到的数据

FIN：断开连接，在传输过程中消耗一个字节的序号。

连接过程

一般说TCP通过三次握手建立连接，具体的握手过程如下图所示：

一般建立连接之前Server端已经出去监听状态，等待客户端发送连接，下面直接从分析3次握手的过程：

  1. 客户端发送SYN连接请求，并且告诉服务端自己的序号为x
  2. 服务端接收到连接请求时，也得向客户端发送SYN连接请求，并且告诉客户端自己的序号为y，同时确认收到客户端的SYN请求，由于SYN请求消耗1个字节，所以ACK为x+1.
  3. 客户端接收到服务端的连接请求后确认ACK序号，没有问题后，就建立连接，并且发送确认信息，序号为x+1，同时确认序号为y+1，也是应为收到了客户端SYN请求，消耗一个字节，所以为y + 1
  4. 服务端收到确认消息后也建立连接

为什么要三次握手

众所周知，数据在网络传输的过程中会丢失会延迟等，如果不使用三次握手建立连接，有如下几种情况：

  • 客户端发送一次连接请求后直接连接

    数据传输存在延迟到达的情况，如果这个时候服务端收到了一个过期的连接请求，直接连接就会白白等待，客户端不会建立连接，直接浪费服务器资源，所以不成立。

  • 服务器接收连接请求后，发送连接请求后直接连接，不需要第三步

    数据传输过程中存在丢失数据的情况，如果服务端发送的数据客户端没有收到，这个时候服务端和客户端都会一直在等待连接，所以必须要应答才能建立连接。

可能会有疑问万一客户端的ACK数据丢失了怎么办呢？

服务端在一定时间内没有收到连接的ACK数据会重试，所以只要有应答确认机制就能保证连接过程，如果重试一段时间还收不到应答，就不再建立连接，因为可能是过期的连接请求数据。

断开过程

一般说断开连接需要四次挥手的过程，具体流程如下所示：

断开连接四次挥手过程如下：

  1. 客户端发送数据完成后，准备断开连接，发送FIN请求seq=x，ack=y
  2. 服务端收到后发送确认数据包，FIN也会消耗一个字节的数据，所以ack=x+1
  3. 当服务端数据传输完成后，发送断开连接请求seq=y
  4. 客户端收到后发送断开请求确认，ack= y + 1
  5. 服务端收到后就关闭连接，客户端也在2ML时间后关闭连接

为什么需要四次挥手断开连接

因为TCP是双通道的传输，客户端发送完数据后服务端可能还需要发送数据，所以不能直接断开，双方各自发送自己的断开请求并且只有收到确认数据报ACK后才能确认，所以总够有四次挥手的过程。

发送断开请求方为啥需要等待2ML时间才能关闭

网络数据传输过程中存在数据延迟的情况，而且最后一次发送ACK确认时服务端可能没有收到会重试，所以这个2ML是数据传输最大的延迟时间，确保能正确关闭连接。

  1. 服务端发送的数据延迟到达，关闭数据报提前到达，直接关闭会丢失数据。
  2. 客户端发送关闭链接确认数据报有可能丢失，如果直接关闭会导致服务端有一个等待和重试的过程，浪费服务端资源。

数据传输注意点

  1. 上面我们看抓包分析工具中seq都是从0开始，其实这个是相对原始seq的相对值，客户端和服务端的seq都是随机生成的，不是从0开始，这样可以防止被伪造的IP发送TCP数据攻击。
  2. ACK数据是不算数据序号和确认序号的。
  3. 上面只是分析了从客户端发送数据到服务端，服务端发送到客户端也是一样的，主要是理解数据序号和确认序号。

窗口管理

窗口主要是解决发送端和接收端处理数据速度不一致，TCP发送和接收数据都是先放到数据缓冲区，如果发送数据比较快，接收端无法及时处理数据，这个时候怎么办？

前面介绍TCP协议头有个字段表示Window Size，这其实是告诉发送者当前还能发送的数据大小也就是接收端缓冲区剩余空间大小，发送端最多只能发送Window Size里面数值大小的字节，当WindowSize = 0的时候，发送端就不能发送数据了，只能定期发送一个试探数据，探测Window Size的大小，而发送数据端一般都是阻塞在写数据的方法中，数据传输的过程举例如下：

拥塞控制

拥塞控制主要原则就是先初始化一个传输数据值窗口，当数据传送快的时候就加大这个窗口的值，当数据传送变慢的时候就减小这个值，所以是一个根据网络情况动态变化的窗口值。

慢开始

刚开始的时候不清楚网络情况，这个时候数据传输窗口值比较小，当能快速接收到确认数据的时候就逐渐加大传输窗口的值，慢慢探测合理的数据传输窗口值。

拥塞避免

当数据传输窗口设置到一定大小的时候，如果接收确认数据的时间慢慢到达一个阀值的时候就认为当前发送数据速度过快，网络情况比较差，降低数据传输窗口值到原来的一半。

问题回答

  1. TCP如何保证传输数据的完整性？

    TCP通过数据确认机制确保数据完整，接收端收到数据的时候给出当前接收到数据的序号给发送端，如果中间出现数据丢失也能通过数据重试机制保证数据传输的完整。

  2. TCP为什么需要3次握手建立连接和4次挥手关闭链接？

    网络中传输数据存在丢失的情况，3次握手主要是确保是正常发送请求的流程，同时确认发送端和接收端的数据传输序号。4次挥手主要是由于TCP是双通道数据传输，所以不能直接关闭，根据数据确认机制，最少需要4次才能准确断开连接。

  3. TCP的传输过程是怎样的，窗口管理是什么机制？

    TCP通过序号也确认号以及应答机制以及重传机制保证数据传输的完整性，窗口主要是为了解决发送端和接收端处理数据速度不一致设计的，当窗口为0的时候就不能发送数据了，只能定时发送探测数据探测窗口大小。

  4. 当读写主机处理数据速度不一致的时候，数据怎么办，会覆盖数据吗？

    通过窗口管理解决，不会发生数据覆盖的情况，发送端会阻塞在写数据的函数。
